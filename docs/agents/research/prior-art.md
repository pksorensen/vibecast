# Research findings: prior art for multi-agent operator harnesses (2026-07-10)

> Generated from the multi-agent research workflow (session 9d7d329a, 2026-07-10). Source of truth for the design docs in this folder.

## Summary

Three mature prior-art lineages exist for abstracting coding agents behind one harness, and they cleanly map to three integration planes. (1) coder/agentapi (MIT, Go) is the TUI-scraping lineage: it runs each agent in an in-memory terminal emulator behind a 2-method AgentIO interface (Write bytes / ReadScreen string), detects state by polling screen snapshots until N consecutive identical frames ("stable" vs "changing"), and reconstructs messages by diffing the screen against a snapshot taken before the last user message, then stripping per-agent chrome (echoed input, input-box box-drawing characters that differ per agent). Its per-agent adapter surface is remarkably small: an AgentType enum plus three per-agent functions (FormatMessage, ReadyForInitialPrompt, input formatting), yet it supports 11+ agents. Critically, agentapi is itself migrating to ACP: its x/acpio package implements the same AgentIO interface over an ACP connection, validating a "thin own interface now, ACP backend later" strategy. (2) Zed's Agent Client Protocol (ACP, Apache-2.0) is the protocol lineage and has clearly won the standardization race: ~35 agents implement it as of mid-2026 — Claude Code (via zed-industries/claude-agent-acp built on the Claude Agent SDK), Codex CLI (zed-industries/codex-acp), Gemini CLI (native), Cursor (native), Copilot CLI (public preview Jan 2026), OpenCode, Goose, Kimi, Qwen, Mistral Vibe, and notably pi (via svkozak/pi-acp). ACP v1 covers everything vibecast's metadata plane needs — session lifecycle (session/new, session/load with history replay, session/resume stabilized Apr 2026, session/list), streamed session/update notifications (agent_message_chunk, agent_thought_chunk, tool_call with id/title/kind/status/locations/rawInput/rawOutput, tool_call_update, plans), client-side permission gating (session/request_permission — the CLIENT decides, perfect for vibecast's dangerous-command guard), and even client-owned terminals (agent executes commands in terminals the client provides — vibecast could back these with its tmux panes). Its gaps: stdio-subprocess only today (HTTP/WebSocket transport is an active RFD), and ACP agents run headless — there is no native agent TUI to stream, which collides with vibecast's core "watch the real terminal" value prop. (3) BloopAI/vibe-kanban (Apache-2.0, Rust) is the headless-JSON lineage: a StandardCodingAgentExecutor trait (spawn / spawn_follow_up(session_id, reset_to_message_id) / normalize_logs / MCP-config injection / capabilities / approvals) implemented per agent over each agent's non-interactive JSON mode — Claude via stream-json plus its bidirectional control protocol (intercepting can_use_tool control requests into an ExecutorApprovalService), Codex via exec --json, and Gemini/Qwen/Copilot via a shared AcpAgentHarness speaking ACP — all normalizing into one NormalizedEntry event model that is nearly identical to vibecast's metadata events.

On conformance testing, no industry-standard "agent harness conformance suite" exists, but two proven patterns cover the need: agentapi's e2e suite drives the real harness end-to-end against a scripted fake agent (a JSON scenario file of {expectMessage, thinkDurationMS, responseMessage} steps, with scenarios for basic flow, thinking, stdin, and state persistence/resume — including an ACP echo variant), and vibe-kanban ships a QaMockExecutor (emits ClaudeJson-format logs plus real file operations for full-stack QA) plus fixture-based unit tests replaying recorded agent JSONL through each normalizer. venikman/ACP-inspector adds protocol-traffic conformance validation for ACP specifically. The briefly-surveyed projects confirm the pattern space: coder/mux abandoned wrapping entirely (own agent loop), happy-coder wraps only claude/codex as `happy claude` for mobile handoff + permission push notifications, omnara pivoted from CLI-scraping to the Claude Agent SDK and normalizes via a hub API, Dagger container-use sidesteps agent abstraction by integrating through MCP (environment plane only), and standardagents/dmux is the closest tmux-native multi-agent multiplexer (supports 11 CLIs including pi) but at README depth appears to do launch-command abstraction only.

## Capabilities

### tui-driving-and-input-injection — full

agentapi: AgentIO{Write([]byte), ReadScreen() string} over an in-memory emulator; messages sent as bracketed paste (\x1b[200~...\x1b[201~); writeStabilize 2-phase send — Phase 1 echo-detection (2s, non-fatal timeout since paste-buffering agents don't echo), Phase 2 write \r and wait for screen change (15s fatal, re-send CR every 3s because e.g. aider processes extra CRs). MessageTypeRaw exists because ESC-prefixed paste sequences CANCEL Claude Code's numbered-choice menus — menus need raw keystrokes.

*Evidence:* scratchpad clone: prior-art/agentapi/lib/screentracker/pty_conversation.go (writeStabilize), lib/httpapi/claude.go (formatPaste comment), lib/httpapi/models.go (MessageTypeRaw)

### agent-state-detection — full

agentapi: screen-snapshot ring buffer polled every SnapshotInterval; status=stable only when all N consecutive snapshots are byte-identical AND initial-prompt-readiness detected AND outbound queue empty; states initializing/changing/stable → API running/stable. Readiness = per-agent 'input box present on screen' check (removeMessageBox variants: generic vs Codex vs opencode vs Amp box-drawing). vibe-kanban avoids the problem: headless processes signal completion by exit + ExecutorExitSignal oneshot. ACP signals via stopReason on session/prompt response.

*Evidence:* prior-art/agentapi/lib/screentracker/pty_conversation.go statusLocked/isScreenStableLocked; lib/msgfmt/agent_readiness.go; vibe-kanban executors/mod.rs ExecutorExitResult

### message-and-event-normalization — full

Three tiers. (1) agentapi terminal tier: agent message = screenDiff(screenBeforeLastUserMessage, current) then per-agent strip of echoed input (fuzzy rune matching tolerating agents rewriting input, e.g. Codex strips ```) and input-box chrome (Gemini/Copilot '╯╰', Cursor '┘└', opencode '┃'+2 lines); SSE events: message_update{id,role,message,time}, status_change{status,agent_type}, screen_update{screen}, agent_error. (2) vibe-kanban structured tier: NormalizedEntry{timestamp, entry_type, content, metadata} with entry types UserMessage/AssistantMessage/ToolUse{tool_name, action_type, status}/Thinking/SystemMessage/ErrorMessage/TokenUsageInfo/NextAction/UserAnsweredQuestions; ToolStatus Created/Success/Failed/Denied{reason}/PendingApproval{approval_id}/TimedOut. (3) ACP protocol tier: session/update discriminated union — agent_message_chunk, agent_thought_chunk, tool_call{toolCallId,title,kind:read|edit|delete|move|search|execute|think|fetch|other,status,content,locations,rawInput,rawOutput}, tool_call_update (partial), plan.

*Evidence:* prior-art/agentapi/lib/msgfmt/msgfmt.go, lib/httpapi/events.go; vibe-kanban crates/executors/src/logs/mod.rs:71-137; acp/docs/protocol/v1/tool-calls.mdx

### per-agent-adapter-surface — full

What actually varies per agent across projects: (a) agentapi TUI plane: FormatMessage(msg,userInput), ReadyForInitialPrompt(screen), input-format quirks — 3 small functions per agent, most share the generic one. (b) vibe-kanban headless plane, trait StandardCodingAgentExecutor: spawn(dir,prompt,env), spawn_follow_up(dir,prompt,session_id,reset_to_message_id,env), spawn_review, normalize_logs(MsgStore,worktree)→JoinHandles, default_mcp_config_path + get_mcp_config (per-agent config dialect + JSON path, e.g. codex 'mcp_servers' TOML vs amp 'amp.mcpServers'), use_approvals(service), apply_overrides(ExecutorConfig), capabilities() (SessionFork/SetupHelper/ContextUsage), get_availability_info(), discover_options() (models, streamed as json_patch), get_preset_options(). Profiles in default_profiles.json: executor × variant (DEFAULT/PLAN/...) with user-editable overrides. (c) ACP plane: capabilities negotiated in initialize (sessionCapabilities.resume/load, promptCapabilities); registry manifest (stabilized Mar 2026) standardizes install/launch metadata so clients don't invent integration metadata.

*Evidence:* prior-art/vibe-kanban/crates/executors/src/executors/mod.rs:220-302, profile.rs; agentapi/lib/msgfmt/*; acp/docs/announcements/acp-agent-registry-stabilized.mdx

### session-resume-and-follow-ups — full

vibe-kanban: session id is extracted by parsing the agent's own event stream (extract_session_id from ClaudeJson user/assistant messages → msg_store.push_session_id) and persisted; follow-up = spawn NEW process with per-agent resume args (claude: --resume <id>, plus --resume-session-at <uuid> to truncate history to a message = rollback support; ACP agents: session/load in AcpAgentHarness with per-agent session_namespace). ACP: session/load (replay full history via session/update stream) vs session/resume (reconnect without replay, stabilized Apr 2026) vs session/list — capability-gated. agentapi resumes only its OWN conversation-display state (atomic-write JSON state file with messages + initialPrompt + initialPromptSent; load gated on agent readiness) — it does not resume the agent's internal session. vibecast's tmux-alive/local-file/server-fetch recovery is more advanced than agentapi's, less structured than ACP's.

*Evidence:* vibe-kanban executors/claude.rs:360-382,891; acp/docs/announcements/session-resume-stabilized.mdx, docs/protocol/v1/session-setup.mdx:190-230; agentapi pty_conversation.go SaveState/loadStateLocked

### approval-and-permission-handling — full

vibe-kanban: ExecutorApprovalService trait {create_tool_approval(tool_name)→approval_id; wait_tool_approval(id,cancel)→Approved|Denied{reason}|TimedOut; question variants}. Wired per-agent: Claude via the stream-json bidirectional control protocol (ProtocolPeer reads control_request can_use_tool from stdout, calls on_can_use_tool, writes control_response to stdin — same mechanism the Claude Agent SDK uses); ACP agents via session/request_permission (client returns selected optionId; options support allow/reject once/always). Timeout maps to denial; auto_approve=true only when no service is registered. agentapi x/acpio Phase 1 auto-approves everything (explicit TODO — a foot-gun to avoid). happy/omnara add the mobile push-notification + remote approve/deny UX layer on top.

*Evidence:* vibe-kanban crates/executors/src/approvals.rs, executors/claude/protocol.rs:107-136, executors/claude/client.rs:60-90; agentapi x/acpio/acpio.go:89-96 (auto-approve); acp tool-calls.mdx Requesting Permission

### acp-protocol-and-ecosystem — full

JSON-RPC 2.0 over stdio subprocess (camelCase props, snake_case discriminators, _meta extensibility); lifecycle initialize→authenticate?→session/new|load|resume→session/prompt→session/update stream→stopReason; client baseline is tiny: session/request_permission only (fs/* and terminal/* optional). ~35 agents today incl. Claude (zed-industries/claude-agent-acp over Claude Agent SDK), Codex (zed-industries/codex-acp), Gemini CLI + Cursor + Copilot CLI native, pi (svkozak/pi-acp), OpenCode, Goose, Kimi, Qwen, Mistral Vibe, OpenHands. Governance active: RFD process, registry + session/list + session/resume + logout stabilized spring 2026; Streamable-HTTP/WebSocket remote transport RFD in flight (HTTP/2 + SSE GET streams + Acp-Connection-Id, or WebSocket upgrade, cookies for affinity). GAPS for vibecast: agents run headless — no native TUI to stream (client renders conversation); stdio-only today; client-owned terminal/* methods let the agent run commands in terminals the CLIENT creates — vibecast could back these with tmux panes so command execution stays visible on stream.

*Evidence:* acp/docs/protocol/v1/* (cloned), docs/get-started/agents.mdx (full agent list), docs/rfds/streamable-http-websocket-transport.mdx; https://agentclientprotocol.com

### terminal-streaming-to-viewers — partial

No prior project combines vibecast's live TUI broadcast with normalized metadata events. agentapi exposes screen_update SSE (plain-text 80-col screen snapshots, no ANSI fidelity) as a debug/preview surface. happy-coder syncs the Claude/Codex terminal to mobile with E2E encryption but wraps only 2 agents and doesn't normalize events. dmux (standardagents/dmux) multiplexes 11 agent CLIs (incl. pi) in tmux/worktrees — closest to vibecast's tmux-ownership model — but is launch-orchestration only. This is vibecast's differentiator: keep tmux/ttyd for pixels, use a separate structured plane for events.

*Evidence:* agentapi lib/httpapi/events.go ScreenUpdateBody; https://github.com/slopus/happy; https://github.com/standardagents/dmux

### conformance-and-verification-testing — partial

Best prior art: agentapi e2e — a scripted fake agent (e2e/echo.go) driven by JSON scenario files [{expectMessage, thinkDurationMS, responseMessage}] run under the REAL harness and asserted via the HTTP SDK; scenario corpus: basic.json, thinking.json, stdin.json, state_persistence*.json (3 resume variants), acp_basic.json (same scenarios through the ACP path — proving one scenario suite can verify two adapter backends). vibe-kanban: QaMockExecutor behind qa-mode feature (emits ClaudeJson-format logs over 10s + performs real file ops, reuses production normalizers) + inline fixture tests replaying recorded agent JSONL through normalize_logs. venikman/ACP-inspector validates/debugs live ACP traffic for protocol conformance. Academic: AgentRFC/AgentConform (prose-spec→executable-test pipelines for MCP/A2A/ACP). No industry-standard 'coding-agent harness conformance suite' exists — the echo-agent + recorded-fixture + traffic-validator triad is the assembled state of the art.

*Evidence:* prior-art/agentapi/e2e/echo.go, e2e/testdata/, e2e/echo_test.go; vibe-kanban executors/qa_mock.rs; https://github.com/venikman/ACP-inspector; https://arxiv.org/html/2603.23801v1

### capability-declaration-and-agent-discovery — full

vibe-kanban: capabilities() enum per executor (SessionFork, SetupHelper, ContextUsage), get_availability_info() (PATH/config-file detection), discover_options() streaming discovered models/modes as JSON patches, executor profiles with variants and permission policies. ACP: initialize capability negotiation (agentCapabilities.sessionCapabilities.resume/load etc. — clients MUST NOT call undeclared methods) + curated registry with shared install/launch manifest. Pattern: capabilities are DECLARED not probed, and every optional adapter feature is capability-gated — exactly what a conformance suite should key off.

*Evidence:* vibe-kanban executors/mod.rs BaseAgentCapability, executor_discovery.rs; acp session-setup.mdx capability checks; acp/docs/get-started/registry.mdx

### environment-isolation-plane — partial

Dagger container-use avoids agent abstraction entirely: it is an MCP server ('container-use stdio') any MCP-capable agent mounts, giving each agent an isolated container + git branch with full command history; per-agent work is just docs for adding the MCP server. coder/mux went the opposite way — gave up on wrapping and built its own agent loop (Claude-Code-inspired UX, plan/exec modes). Lesson: the environment/isolation plane can be agent-agnostic via MCP and should stay out of the AgentAdapter interface.

*Evidence:* https://github.com/dagger/container-use; https://github.com/coder/mux README

### remote-control-and-event-hub — partial

omnara: agents→REST API→Postgres→push→mobile/web→responses relayed back; supports Claude Code/Codex/n8n/GH Actions; NOTABLE: deprecated its Claude-Code CLI-scraping wrapper in favor of the Claude Agent SDK (the scraping→SDK pivot everyone eventually makes). happy-coder: `happy claude`/`happy codex` wrapper, E2E-encrypted device sync, push on permission-request, seamless local↔remote input handoff (keyboard on desktop reclaims control) — a UX pattern worth copying for vibecast's viewer-interaction story.

*Evidence:* https://github.com/omnara-ai/omnara; https://github.com/slopus/happy

## Recommended integration strategy

Recommended design for vibecast's AgentAdapter, synthesized from what survived contact with reality in these projects:

(a) ADAPTER INTERFACE SHAPE — split into two planes and keep the per-agent surface tiny. Presentation plane (vibecast already owns): the agent's real TUI in tmux, streamed via ttyd — no abstraction needed, it's bytes. Control/metadata plane (the AgentAdapter): model it on vibe-kanban's trait, not agentapi's screen-scraper, because vibecast already renders the TUI and only needs structured events. Per-agent members that prior art proved necessary: BuildLaunchCommand(workdir, prompt, opts) and BuildResumeCommand(sessionID, opts) (resume flags differ: claude `--resume <id>` / `--resume-session-at <uuid>`, codex `exec resume`, ACP `session/load|resume`); ExtractSessionID(eventStream) (session ids only exist inside the agent's own output — vibe-kanban parses them from the stream and stores them); MetadataSource — the per-agent strategy for structured events (Claude: hooks as today, or stream-json control protocol; Codex: exec --json / notify; Gemini/pi/Codex/Copilot: ACP); NormalizeEvent(raw) → vibecast metadata event; ApprovalHook — how permission requests surface (Claude control-protocol can_use_tool, ACP session/request_permission, or none in TUI mode where vibecast's existing command guard applies); Capabilities() declaration (resume, fork, toolEvents, contextUsage, approvals) — declared, never probed, and capability-gated everywhere like ACP does; DetectReady(screen) and input quirks (bracketed-paste vs raw keys) ONLY for agents driven through the TUI — copy agentapi's three-function pattern (FormatMessage/ReadyForInitialPrompt/input format) if you must scrape, and note most agents shared the generic implementation. Keep MCP/config injection (vibe-kanban's get_mcp_config path-dialect table) as an optional member — you'll want it the day you inject vibecast's own MCP tools.

(b) EVENT NORMALIZATION — adopt vibe-kanban's NormalizedEntry shape as the internal event model ({timestamp, entry_type, content, metadata} with UserMessage/AssistantMessage/ToolUse{tool_name, action_type, status}/Thinking/SystemMessage/Error/TokenUsage) but name fields with ACP's vocabulary (tool_call with toolCallId/title/kind[read|edit|execute|...]/status[pending|in_progress|completed|failed], agent_message_chunk, agent_thought_chunk, plan, stopReason) so a later ACP backend is a field-mapping no-op. Two invariants prior art converged on: tool events are keyed by a stable toolCallId with partial updates (ACP tool_call_update sends only changed fields), and approval state is part of tool status (PendingApproval{approval_id} → Denied{reason}/TimedOut, timeout = denial).

(c) ACP: DON'T adopt as the only interface; DO make it the flagship backend. Both deep-dive projects independently chose this hybrid: agentapi built x/acpio implementing its existing AgentIO over ACP, and vibe-kanban runs Gemini/Qwen/Copilot through one shared AcpAgentHarness while keeping bespoke executors for Claude/Codex. Reasons ACP can't be the whole story for vibecast: ACP agents run headless (no native TUI to broadcast — the client renders), transport is stdio-only today (remote transport is a pending RFD), and vibecast's richest integration (Claude hooks + real TUI + tmux resume) predates and exceeds what claude-agent-acp exposes. So: define the thin Go AgentAdapter interface, ship ClaudeTUIAdapter (current hooks path, unchanged) first, then one ACPAdapter that instantly onboards Gemini CLI, Codex (codex-acp), pi (pi-acp), Copilot, OpenCode, Kimi, Qwen — that single adapter is the "mostly-mechanical onboarding" story. Consider implementing ACP's client-side terminal/* methods backed by tmux panes so ACP agents' command executions stay visible on stream, partially restoring the broadcast value prop for headless agents. Watch the transports RFD; if vibecast ever needs remote agents, don't invent a transport.

(d) CONFORMANCE TESTING — assemble the proven triad: (1) Scenario suite against a scripted fake agent (copy agentapi's echo agent: JSON scenarios of {expectMessage, thinkDuration, responseMessage} run under the REAL vibecast harness in tmux, asserted via vibecast's own metadata events) — agentapi even proved the same scenario corpus can validate two different adapter backends (basic.json vs acp_basic.json). Vibecast's scenario list should be: launch→ready, prompt→completion event, tool-call lifecycle, permission-request→approve and →deny, dangerous-command guard trip, crash→resume (all three recovery paths), terminal resize. (2) Recorded-fixture replay: capture each real agent's raw event stream once, commit as fixtures, replay through NormalizeEvent in unit tests (vibe-kanban's pattern — catches agent version churn cheaply in CI without API keys). (3) A QaMock adapter emitting the normalized format directly for full-stack e2e of the server/viewer pipeline (vibe-kanban's QaMockExecutor). Gate each capability claim on a scenario: an adapter declaring `resume` must pass the resume scenarios — this makes the conformance doc the onboarding checklist.

(e) PITFALLS TO AVOID — all empirically hit by these projects: ESC-prefixed bracketed-paste sequences cancel Claude Code's selection menus (agentapi needed a raw-keystroke escape hatch; vibecast's prompt injection must special-case menus); some agents process stray carriage returns as input (aider) while others need CR retries — agentapi's 3s-CR-retry/15s-fatal writeStabilize numbers are battle-tested; input-echo removal is inherently fragile (agents rewrite your input — Codex strips ```; every agent draws a different input box: ╯╰ vs ┘└ vs ┃) — avoid screen-scraping for metadata entirely, scrape only if an agent has no structured mode; screen-stability state detection breaks on spinners/timers and on every agent UI redesign — prefer process-exit + structured stop events (vibe-kanban) or ACP stopReason; auto-approving permissions in a first ACP integration (agentapi's x/acpio Phase 1) silently disables your guard — wire request_permission to the vibecast guard from day one and treat approval timeout as denial; headless/bypass modes may never emit permission requests, so keep vibecast's command guard OUTSIDE the agent as defense in depth; don't store conversation state yourself as the resume mechanism when the agent owns the session (agentapi's state file only restores display history — vibe-kanban's "extract the agent's own session id and re-invoke with resume flags" is the correct model, matching vibecast's existing claudeSessionId flow); and expect the scraping→SDK/protocol pivot (omnara deprecated its CLI wrapper; agentapi is building acpio) — design so the TUI-scrape path is the fallback, not the foundation.

## Open questions

- How mature are the individual ACP adapters vibecast would rely on — does codex-acp expose approvals/session-resume with parity to codex exec, and is svkozak/pi-acp (third-party) maintained enough to bet on, or should pi get a native adapter?
- Can vibecast implement ACP's client-side terminal/create|output|kill methods backed by tmux panes so headless ACP agents' command executions render on the broadcast — and is that enough 'terminal' to preserve the viewer experience, or do headless agents need a conversation-renderer view in the viewer instead?
- Claude Code specifics vibe-kanban depends on that we would too: are --resume-session-at and the stream-json bidirectional control protocol (can_use_tool control requests) stable public surface, and do they conflict with vibecast's existing hooks-based metadata (double-reporting of tool calls)?
- ACP remote-transport RFD (Streamable HTTP/WebSocket) timing — if it lands, does vibecast's ws-relay want to speak it natively so remote/containerized agents plug in without stdio piping?
- Do Gemini CLI / Codex have hook systems comparable to Claude's (for TUI-mode metadata without ACP), or is TUI-mode for non-Claude agents necessarily metadata-blind (screen-scrape or nothing)? Not verified this pass.
- Licensing is clean for pattern-copying (agentapi MIT, vibe-kanban Apache-2.0, ACP Apache-2.0 with Go SDK coder/acp-go-sdk available) — but if we vendor agentapi's msgfmt heuristics or use coder/acp-go-sdk, confirm attribution requirements in the vibecast open-source submodule.
- dmux (standardagents/dmux) was only surveyed at README level — its tmux-per-agent launch matrix for 11 CLIs (incl. pi) may contain ready-made launch/resume flag tables worth mining before writing our own.
