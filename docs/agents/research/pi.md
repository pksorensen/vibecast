# Research findings: pi-coding-agent (0.73.1 local / 0.80.6 latest, empirically verified 2026-07-10)

> Generated from the multi-agent research workflow (session 9d7d329a, 2026-07-10). Source of truth for the design docs in this folder.

## Summary

pi (Mario Zechner's pi-coding-agent) is an excellent integration target for vibecast — arguably easier than Claude Code, because its extension system is a strict superset of Claude Code's hooks: TypeScript modules loaded in-process (via -e flag or ~/.pi/agent/extensions/) that subscribe to a complete lifecycle event stream (session_start/shutdown, before_agent_start with the user's prompt text, agent_start/end, turn_start/end, message_*, tool_execution_start/update/end, tool_call, tool_result, compaction events) and can BLOCK or mutate tool calls before execution ({block:true, reason}), rewrite the system prompt per turn, and run arbitrary Node code (fs + fetch verified unrestricted — no sandbox). I verified all of this empirically end-to-end on the installed 0.73.1 by registering a mock provider (no API key needed) and running a vibecast-style probe extension in both --mode json and the interactive TUI inside tmux: events logged, dangerous `rm -rf` blocked (surfaces as isError toolResult with the block reason), HTTP POST from the extension delivered, prompt captured, session file + id observed, resume by partial UUID confirmed, and tmux send-keys + Enter submission works with zero first-run dialogs.

Two strategic findings need attention. First, the project moved: @mariozechner/pi-coding-agent is deprecated at 0.73.1 (what's installed); the live package is @earendil-works/pi-coding-agent, latest 0.80.6 (2026-07-09), which requires Node >=22.19 (a legacy-node20 dist-tag pins 0.74.2 — this devcontainer runs Node 20.20.2). The newer versions add exactly the primitives vibecast wants: --session-id <id> ("use exact project session ID, creating it if missing" — the perfect crash-resume handle, v0.76+), agent_settled event (v0.80.4), --name for session display name, and a project-trust prompt (v0.79+) that gates .pi/ project-local resources in interactive mode — bypass with --approve / defaultProjectTrust:"always" / pre-seeding ~/.pi/agent/trust.json. Second, auth on a keyless machine is solvable three ways: Claude Pro/Max subscription OAuth via /login (verified: PKCE URL + localhost callback + paste-redirect-URL fallback for headless boxes; note the docs say third-party-harness subscription use bills as per-token "extra usage", not plan limits), a gateway/proxy via pi.registerProvider("anthropic",{baseUrl}) in the same vibecast extension (pks-agent-gateway fits), or writing ~/.pi/agent/auth.json directly. Auth failure is cleanly detectable: headless prints "No API key found for the selected model." (exit 1); the TUI shows "Warning: No models available. Use /login".

pi deliberately ships no MCP, no permission popups, no sub-agents ("primitives, not features") — the permission-gate extension pattern is the guard, and a community pi-mcp-adapter package exists if MCP is ever needed. ACP was requested (issue #175) and not implemented; RPC mode (bidirectional JSONL over stdio with prompt/steer/abort/get_state/get_session_stats commands and an extension-UI sub-protocol) plus the SDK are the embedding APIs, but for vibecast's TUI-in-tmux model the extension is the right event channel since json/rpc modes replace the interactive UI.

## Capabilities

### launch-flags — full

--system-prompt <text> (replace), --append-system-prompt <text|file> (repeatable) for the station prompt; initial prompt as positional argv in interactive mode — verified it auto-submits and runs immediately on startup (also @files, multiple messages); -p/--print for one-shot; --extension/-e <path> repeatable + auto-discovery dirs ~/.pi/agent/extensions/*.ts and .pi/extensions/*.ts; --no-extensions/-ne keeps explicit -e working while disabling discovery; config dir override via PI_CODING_AGENT_DIR (default ~/.pi/agent), session dir via --session-dir / PI_CODING_AGENT_SESSION_DIR; working dir = process cwd (no --cwd flag — launch in target dir; sessions are keyed by cwd); tool control --tools/-t allowlist, --no-tools, --no-builtin-tools; latest (@earendil-works ≥0.77) adds --exclude-tools/-xt, --name/-n <session display name>, --approve/-a and --no-approve/-na (project trust), --session-id <id>. Extensions can also register their own CLI flags (pi.registerFlag).

*Evidence:* pi --help output (0.73.1); verified argv prompt auto-run in tmux TUI test; latest flags confirmed in earendil-works/pi src/cli/args.ts and changelog

### tui-in-tmux — full

Runs cleanly in tmux (tested tmux 3.3a, 200x50 pane): full-screen TUI with editor, footer (cwd, tokens ↑↓, context %/window, model). tmux send-keys 'text' then Enter submits — verified the prompt reached the agent (captured by extension before_agent_start). Plain Enter submits; Shift+Enter = newline needs tmux extended-keys (set -g extended-keys on; format csi-u) else pi prints a cosmetic warning — submission itself unaffected. NO first-run onboarding/trust dialogs on 0.73.1; goes straight to editor. Keyless first run shows 'Warning: No models available. Use /login...' but stays usable. Latest versions (≥0.79) add a project-trust prompt when the project has .pi/ resources (settings/extensions/skills) and no saved decision — interactive-only; bypass with --approve/--no-approve per run, global defaultProjectTrust setting (ask|always|never), pre-seeded ~/.pi/agent/trust.json (keyed by canonical dir), or a global extension answering the project_trust event. AGENTS.md/CLAUDE.md load regardless of trust. docs/tmux.md is a dedicated page.

*Evidence:* Empirical: tmux new-session + capture-pane runs in scratchpad pi-probe; send-keys prompt captured in /tmp/pi-probe-events.log; trust system from earendil-works/pi docs/security.md

### lifecycle-events — full

Extension events (in-process TS, pi.on): session_start {reason: startup|reload|new|resume|fork, previousSessionFile?} with ctx.sessionManager.getSessionFile()/getSessionId(); session_shutdown {reason: quit|reload|new|resume|fork} — fires on Ctrl+C/D, SIGHUP, SIGTERM per docs lifecycle diagram; agent_start / agent_end {messages} once per user prompt (= 'completion' signal); turn_start / turn_end {turnIndex, message, toolResults} per LLM turn; agent_settled added v0.80.4; session_before_switch/fork/compact/tree cancelable pre-events; input event (raw text + source interactive|rpc|extension). Same agent_/turn_ events also stream as JSONL on stdout in --mode json and --mode rpc. No shell-command hook system — the TypeScript extension IS the hook mechanism (runs in pi's process, can fetch() out).

*Evidence:* docs/extensions.md lifecycle diagram + empirical probe log: session_start→before_agent_start→agent_start→turn_start/end×2→agent_end→session_shutdown all observed in order

### tool-call-events — full

Per-tool-call extension events with full payloads: tool_execution_start {toolCallId, toolName, args} → tool_call {toolName, toolCallId, input (MUTABLE — patch args in place; return {block:true, reason} to deny)} → tool_execution_update {partialResult, accumulated} → tool_result {content, details, isError — return partial patches to modify} → tool_execution_end {result, isError}. Parallel-tool-mode ordering caveats documented (start in source order, end in completion order). Identical events emitted as JSONL in --mode json / rpc (but those modes replace the TUI). Third option: session-file tailing (~/.pi/agent/sessions/...jsonl gets toolCall content blocks in assistant messages + toolResult messages) — post-hoc only, no pre-execution visibility. Mechanism ranking for vibecast: extension (in-process, pre+post+block) > rpc/json JSONL (pre+post, no TUI) > session tail (post only).

*Evidence:* Empirical: all events captured in probe log incl. blocked rm -rf (tool_execution_end isError:true) and safe echo (tool_result content 'hello-from-mock'); docs/extensions.md Tool Events section

### prompt-capture — full

before_agent_start event carries event.prompt (user's submitted text) + event.images + systemPromptOptions — fires for typed TUI prompts, argv-injected initial prompts, and RPC prompts alike (verified for the first two). input event sees raw pre-expansion text with source: 'interactive'|'rpc'|'extension' and can transform/handle it. message_start/message_end with role:'user' carry the stored UserMessage. Session JSONL persists every user message. In RPC mode the client sends the prompt itself so capture is trivial.

*Evidence:* Empirical: probe log shows {prompt:'initial prompt injected via argv'} and {prompt:'second prompt via send-keys'} from before_agent_start

### session-resume — full

Sessions auto-save to ~/.pi/agent/sessions/--<cwd-with-slashes-as-dashes>--/<ISO-timestamp>_<uuid>.jsonl (JSONL v3, tree via id/parentId, header line {type:'session',version:3,id:<uuid>,cwd}). sessionId = the UUID (in filename + header + extension ctx.sessionManager.getSessionId() + RPC get_state). Discovery: extension session_start handler can POST sessionFile+sessionId to vibecast (verified); or SessionManager.list(cwd); or ls the cwd-derived dir. Resume: -c/--continue (most recent for cwd), -r picker, --session <path|partial-uuid> (verified: appends to same file/id), --fork <path|id> (new file), --no-session (ephemeral). Latest (≥0.76): --session-id <id> = 'use exact project session ID, creating it if missing' — ideal crash-resume: vibecast mints the UUID up front, relaunches with the same flag. Override storage root with --session-dir/PI_CODING_AGENT_SESSION_DIR. Sessions are per-cwd, so resume must run in the same working directory (or pass --session with full path).

*Evidence:* Empirical: session file created at documented path; --session 019f4cf7 (partial uuid) resumed same id/file; --session-id confirmed in earendil-works/pi src/cli/args.ts help text; docs/session-format.md

### permissions-guard — full

No built-in permission system — pi deliberately YOLOs all tool calls (README: 'no permission popups'; docs/security.md: no sandboxing, tools run with user perms). The guard is an extension: tool_call handler returns {block:true, reason} → tool never executes, LLM sees an isError toolResult containing the reason (verified: 'vibecast guard: dangerous command blocked'), agent continues. Input is mutable pre-execution (can rewrite commands). Interactive approval: ctx.ui.confirm/select inside the handler (ships as examples/extensions/permission-gate.ts with rm -rf/sudo/chmod-777 patterns); ctx.hasUI is false in -p/json modes (block-by-default pattern documented) and true in RPC (dialogs travel over the extension_ui_request/response sub-protocol, with optional timeout auto-resolve). tool_result hook can additionally censor/modify outputs. Latest adds project-trust gating but that's resource loading, not tool calls.

*Evidence:* Empirical: rm -rf blocked, echo allowed, both in json mode and interactive TUI; permission-gate.ts read from installed package examples/

### auth-onboarding — full

Interactive /login → 'Use a subscription' (Anthropic Claude Pro/Max, ChatGPT Plus/Pro Codex, GitHub Copilot) or 'Use an API key' (per-provider). Claude Pro/Max OAuth verified: PKCE authorize URL on claude.ai/oauth/authorize + localhost callback server + 'If the browser is on another machine, paste the final redirect URL here' — fully headless-workable. IMPORTANT: docs/providers.md says third-party-harness subscription usage draws from Anthropic 'extra usage' billed per token, NOT plan limits. Tokens/keys stored in ~/.pi/agent/auth.json (0600; auto-refresh; wins over env vars); key field accepts literal | ENV_VAR_NAME | '!shell-command'. Env vars: ANTHROPIC_API_KEY, OPENAI_API_KEY, GEMINI_API_KEY (provider 'google' = default provider), plus ~20 more (see envMap in packages/ai/src/env-api-keys.ts); also Azure OpenAI (AZURE_OPENAI_*), Bedrock (AWS_*), Vertex (ADC). Custom base URL / gateway: pi.registerProvider('anthropic',{baseUrl}) from an extension (keeps all models), or models.json, or full custom provider w/ own OAuth. Waiting-for-login detection: TUI prints 'Warning: No models available. Use /login to log into a provider via OAuth or API key.'; headless -p/json exits 1 with stderr 'No API key found for the selected model.'; RPC prompt returns {success:false, error:'No API key found...'}. Resolution order: --api-key > auth.json > env > models.json.

*Evidence:* Empirical: /login flow driven via tmux send-keys through provider list to live OAuth URL; keyless error shapes captured for json/rpc/TUI; docs/providers.md tables

### headless-mode — full

Three non-interactive surfaces: (1) -p/--print: process prompt(s), print text, exit; reads piped stdin into prompt; exit 0 on success, 1 on auth/error (verified). (2) --mode json [-p] <prompt>: JSONL event stream on stdout — first line session header, then agent_start/turn_start/message_start/message_update{assistantMessageEvent deltas: text_delta, thinking_delta, toolcall_start/delta/end, done, error}/message_end/tool_execution_*/turn_end/agent_end{messages}, plus queue_update, compaction_start/end, auto_retry_start/end, extension_error. Caveat: in one run with session persistence + extensions, agent_end wasn't flushed to stdout before exit (it did fire to the extension); with --no-session it appeared — treat process-exit as completion backstop. (3) --mode rpc: bidirectional strict-LF JSONL over stdin/stdout; commands: prompt (with streamingBehavior steer|followUp), steer, follow_up, abort, new_session, get_state (isStreaming, sessionFile, sessionId, model), get_messages, set_model, get_session_stats (tokens/cost/contextUsage), compact, bash, switch_session, fork, clone, get_last_assistant_text, set_session_name, get_commands (+ get_entries/get_tree on ≥0.80.3); optional id correlation; extensions' UI dialogs surface as extension_ui_request/response. SDK (createAgentSession) for Node embedding; openclaw/openclaw is a real-world SDK consumer.

*Evidence:* Empirical json + rpc runs (json-mock.out, rpc-noauth.out in scratchpad/pi-probe); docs/json.md + docs/rpc.md read in full

### mcp — partial

No built-in MCP client or server — an explicit design decision (README: 'No MCP. Build CLI tools with READMEs, or build an extension that adds MCP support', linking Mario's 'What if you don't need MCP' post; usage.md lists MCP among deliberately-excluded features). Community fills the gap: pi-mcp-adapter package (pi install npm:pi-mcp-adapter) bridges MCP servers to pi custom tools; awesome-pi.site catalogs it. Writing one is easy: extension registers tools via pi.registerTool that proxy an MCP client. No MCP server mode at all — pi is exposed programmatically via RPC/SDK instead. ACP (Zed's Agent Client Protocol): requested in issue #175, closed without implementation; not supported.

*Evidence:* grep of installed README/docs ('No MCP'); web: awesome-pi.site extensions list, github.com/badlogic/pi-mono issue #175 fetch

### context-events — full

Compaction: auto (contextTokens > contextWindow - reserveTokens, default reserve 16384, keepRecentTokens 20000, configurable) or /compact — emits compaction_start {reason: manual|threshold|overflow} and compaction_end {reason, result:{summary, firstKeptEntryId, tokensBefore}, aborted, willRetry, errorMessage?} in json/rpc streams; extensions get cancelable session_before_compact (can supply custom summary) + session_compact; CompactionEntry persisted in session JSONL. Token usage: every assistant message carries usage {input, output, cacheRead, cacheWrite, totalTokens, cost{...total}}; extension ctx.getContextUsage() → {tokens, contextWindow, percent}; RPC get_session_stats → aggregate tokens/cost + contextUsage (percent null right after compaction until fresh usage); TUI footer shows ↑↓ tokens + context %/window live (observed '↑20 ↓10 0.0%/128k'). set_auto_compaction RPC / compaction.enabled setting to disable.

*Evidence:* docs/compaction.md, docs/json.md event types, docs/rpc.md get_session_stats; footer observed in tmux capture

## Recommended integration strategy

Recommended architecture: vibecast ships a single ~100-line "vibecast.ts" pi extension and treats it exactly like its Claude Code hooks bundle. (a) LAUNCH: spawn `pi` in the vibecast-owned tmux pane with cwd = workspace, args: `--append-system-prompt "$STATION_PROMPT" -e /opt/vibecast/pi/vibecast.ts --name "<task title>" "<initial prompt>"` — the positional prompt auto-submits on startup (verified). Pass VIBECAST_ENDPOINT/VIBECAST_STREAM_ID via env (the extension reads process.env). On @earendil-works ≥0.76 add `--session-id <vibecast-generated-uuid>` and `--approve` (or pre-seed ~/.pi/agent/trust.json / set defaultProjectTrust) so no trust dialog ever blocks the TUI; set PI_SKIP_VERSION_CHECK=1 for deterministic startup. (b) EVENTS — ranking: (1) the vibecast extension POSTing to vibecast's local HTTP listener (same pattern as the Claude Code hook command): verified full fidelity — session_start (carries sessionFile + sessionId → replaces Claude's session-id hook), before_agent_start (user prompt text), tool_execution_start/tool_call (pre), tool_result/tool_execution_end (post), turn_end, agent_end (completion), session_shutdown (end; fires on SIGTERM/SIGHUP) — and it works while the interactive TUI renders, which --mode json/rpc do not (they replace the TUI, so they're only right if vibecast ever renders its own UI); (2) --mode rpc for a future non-TUI runner (steer/abort/get_session_stats are nice); (3) session-JSONL tailing only as crash-recovery backfill (no pre-tool events, no blocking). (c) RESUME: vibecast mints the session UUID and relaunches with `--session-id <uuid>` (≥0.76); on the currently-installed 0.73.1 fall back to capturing sessionId from the extension's session_start POST and relaunching `--session <uuid>` (verified with partial UUID), with `-c` as last resort — same cwd required since sessions are keyed by directory. (d) COMPLETION: agent_end from the extension (one per user prompt; agent_settled on ≥0.80.4 is even stricter); session_shutdown = broadcaster end. (e) GUARD: port vibecast's dangerous-command regexes into the extension's tool_call handler → `{block:true, reason}` (verified: tool never runs, reason lands in the transcript), optionally ctx.ui.confirm for on-stream operator approval — strictly stronger than Claude Code's PreToolUse since input is also mutable. (f) AUTH on this keyless machine: preferred = route pi through pks-agent-gateway by having the same vibecast extension call pi.registerProvider("anthropic", { baseUrl: process.env.VIBECAST_ANTHROPIC_PROXY }) and writing auth.json `{"anthropic":{"type":"api_key","key":"GATEWAY_TOKEN_ENV"}}`; alternative = one-time interactive `/login` → Claude Pro/Max OAuth (verified headless-friendly via paste-redirect-URL; tokens persist + auto-refresh in ~/.pi/agent/auth.json, but NOTE Anthropic bills third-party-harness subscription use as per-token extra usage, not plan quota). Detect "waiting for login" by grepping the pane for "No models available. Use /login" (TUI) or stderr "No API key found for the selected model." (headless). Version pin: install @earendil-works/pi-coding-agent@0.80.x on Node ≥22.19 (Node 20 machines cap at legacy-node20 = 0.74.2, which lacks --session-id).

## Open questions

- Node runtime: target machines must move to Node >=22.19 to run @earendil-works/pi-coding-agent >=0.75 (this devcontainer has 20.20.2, capped at legacy-node20 0.74.2 which lacks --session-id, agent_settled, and trust flags) — is a Node bump acceptable in the vibecast runner image?
- agent_end occasionally not flushed to --mode json stdout before process exit when session persistence + extensions are active (reproduced once; fine with --no-session) — irrelevant for the extension path but worth an upstream issue if json mode is ever used.
- Does `pi update` on an old-scope @mariozechner install migrate to the @earendil-works scope, or silently stay at 0.73.1? Untested — safest is explicit npm install of the new scope.
- Claude Pro/Max OAuth token refresh in long-lived unattended sessions: auto-refresh is documented (auth.json), but not exercised here; also confirm Poul accepts that subscription use through pi bills as Anthropic per-token 'extra usage' rather than plan quota (docs/providers.md).
- Project-trust prompt (>=0.79): confirm --approve fully suppresses it in every interactive launch path (couldn't test locally on 0.73.1), and whether vibecast should pre-seed ~/.pi/agent/trust.json instead so operators can't get stuck.
- Terminal rendering under ttyd/xterm.js at viewer-driven resize: pi's TUI uses differential rendering (pi-tui) and behaved at 200x50 in tmux, but resize-storm behavior (vibecast's docs/terminal-sizing.md pipeline) is untested.
- If vibecast wants mid-run steering/abort from web viewers (like Claude Code's tmux injection), pi offers cleaner primitives: tmux send-keys works today; the extension could also expose pi.sendUserMessage(text, {deliverAs:'steer'}) via a tiny local HTTP endpoint inside the extension — worth deciding which to standardize on.
