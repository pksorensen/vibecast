# Research findings: OpenAI Codex CLI (0.142.5, empirically verified 2026-07-10)

> Generated from the multi-agent research workflow (session 9d7d329a, 2026-07-10). Source of truth for the design docs in this folder.

## Summary

Codex CLI 0.142.5 (npm @openai/codex; latest 0.144.1) is now a near drop-in integration target for vibecast: it has a Claude Code-style lifecycle hooks system (feature flag `hooks` = stable + enabled) with SessionStart, UserPromptSubmit, PreToolUse, PermissionRequest, PostToolUse, PreCompact, PostCompact, SubagentStart/Stop and Stop events. Command hooks receive a JSON payload on stdin that is almost field-for-field identical to Claude Code's (session_id, turn_id, transcript_path, cwd, hook_event_name, model, permission_mode, tool_name, tool_input, tool_use_id, tool_response, prompt, last_assistant_message), and a PreToolUse hook can deny a tool call via `hookSpecificOutput.permissionDecision: "deny"` (empirically verified in the interactive TUI: Codex rendered "PreToolUse hook (blocked)" and fed the reason back to the model). Hooks fire in both `codex exec` and the interactive TUI. All of this was verified live on the logged-in local install, including a TUI-in-tmux run driven entirely by tmux send-keys.

The remaining vibecast capabilities map cleanly: station system prompt appends via `-c developer_instructions="..."` (verified with `codex debug prompt-input` to land in a developer-role message without replacing base instructions) plus AGENTS.md; initial prompt as a CLI positional arg auto-submits in the TUI; session id is a UUIDv7 surfaced by the SessionStart hook (with transcript_path pointing at the rollout JSONL under ~/.codex/sessions/YYYY/MM/DD/), and crash-resume is `codex resume <uuid>` (or `codex exec resume <id>`); turn completion is observable via both the Stop hook and the `notify` config (argv JSON `agent-turn-complete` with last-assistant-message, verified in TUI). Headless mode (`codex exec --json`) emits a JSONL stream (thread.started/turn.started/item.completed/turn.completed), though 0.142.5 empirically omitted command_execution items that the docs promise. There is also an experimental app-server (JSON-RPC 2.0, thread/turn/item primitives, server-initiated approval requests) for deeper control.

Two operational gotchas: (1) non-managed hooks require an interactive one-time trust review ("Hooks need review" dialog) unless launched with `--dangerously-bypass-hook-trust` (works, but injects warning items) or delivered as auto-trusted managed hooks via requirements.toml; (2) the guard is mechanically solid but the model routes around naive filters — when the hook denied `rm`, the model deleted the file with `find -delete`, so vibecast's guard must evaluate semantics, not substrings. No SessionEnd hook exists; synthesize session end from process/pane exit. All probe artifacts (help texts, JSONL captures, hook payloads, notify payloads, notes) are in /tmp/claude-1000/-workspaces-agentic-live-www/9d7d329a-86e0-4692-ad5a-b3b629077a7d/scratchpad/codex-probe/.

## Capabilities

### launch-flags — full

Interactive: `codex [OPTIONS] "initial prompt"` — positional prompt auto-submits in the TUI (verified). Working dir: `-C/--cd <DIR>` + `--add-dir <DIR>` for extra writable roots. System-prompt APPEND: `-c developer_instructions="<station prompt>"` injects into a developer-role message alongside permissions instructions WITHOUT replacing base instructions (verified via `codex debug prompt-input`); AGENTS.md at repo root is injected as a user message (project_doc_max_bytes / project_doc_fallback_filenames configurable); `model_instructions_file` REPLACES base instructions (avoid; successor of experimental_instructions_file). Config: `-c key=value` dotted-path TOML overrides of ~/.codex/config.toml; `-p/--profile` layers $CODEX_HOME/<name>.config.toml; CODEX_HOME env relocates the whole config dir (incl. auth); project layer `<repo>/.codex/config.toml` + hooks.json + .rules (loads only if project trusted). Plugins: `codex plugin add/list/marketplace` (~/.codex/plugins). Also: `-m` model, `-i` images, `--enable/--disable <FEATURE>`, `--no-alt-screen` inline (scrollback-preserving) TUI mode — interesting for streaming.

*Evidence:* codex --help captured to scratchpad/codex-probe/help-main.txt; prompt composition verified with `codex debug prompt-input` (scratchpad/codex-probe/prompt-input.json): developer_instructions probe text in developer message, AGENTS.md and CLI prompt as user messages.

### tui-in-tmux — full

Ratatui alt-screen TUI runs cleanly in tmux (tested 180x45); `--no-alt-screen` gives inline mode. tmux send-keys works: send text, sleep ~1s, send Enter (verified two prompt injections). Dialogs are numbered menus answerable via send-keys ("2"+Enter, "y"). First-run dialogs observed in a brand-new dir: (1) hooks-review gate when untrusted hooks present: "Hooks need review / N hooks are new or changed / 1. Review hooks 2. Trust all and continue 3. Continue without trusting"; (2) command approval dialog: "Would you like to run the following command? … 1. Yes (y) 2. Yes, and don't ask again for commands that start with `touch` (p) 3. No (esc)". NO folder-trust dialog appeared in 0.142.5 for an untrusted new dir (unlike older versions) — projects."<path>".trust_level="trusted" in config.toml pre-trusts a folder and can be passed as a -c override. A bubblewrap warning banner shows in devcontainers (uses bundled bwrap). Initial CLI-arg prompt auto-submitted immediately after the hooks dialog was answered.

*Evidence:* Live tmux session `codexprobe` in scratchpad/codex-probe/repo: captured panes showing hooks-review dialog, banner, PING reply to injected prompt, approval dialog, and hook-deny feedback.

### lifecycle-events — full

Three mechanisms. (1) Hooks (feature `hooks` stable+enabled): SessionStart (payload has `source`; fires on start AND resume), UserPromptSubmit, Stop (payload: last_assistant_message, stop_hook_active — fires per turn completion), SubagentStart/SubagentStop, PreCompact/PostCompact, PermissionRequest. No SessionEnd hook — synthesize session end from codex process / tmux pane exit. Config in ~/.codex/hooks.json, [hooks] in config.toml, or <repo>/.codex/hooks.json; payload as JSON on stdin. (2) notify = ["/path/prog"] (machine-level config only; project layer cannot set it): fires ONLY `agent-turn-complete`, passes ONE JSON argv: {"type":"agent-turn-complete","thread-id","turn-id","cwd","client":"codex-tui","input-messages":[…],"last-assistant-message"} — verified firing in interactive TUI per turn. (3) exec --json: thread.started{thread_id}/turn.started/turn.completed{usage}/turn.failed. Rollout JSONL also logs event_msg task_started{turn_id}/task_complete/turn_aborted.

*Evidence:* scratchpad/codex-probe/hook-events.jsonl (SessionStart/UserPromptSubmit/Stop captured in both exec and TUI runs); notify-events.log (two agent-turn-complete payloads from TUI); exec-events.jsonl; docs at developers.openai.com/codex/hooks (→ learn.chatgpt.com/docs/hooks).

### tool-call-events — full

Best mechanism: PreToolUse/PostToolUse hooks — fire in BOTH exec and interactive TUI around every tool call. Stdin JSON payload (verified): {session_id, turn_id, transcript_path (absolute path to the session rollout JSONL), cwd, hook_event_name, model, permission_mode ("default" TUI / "bypassPermissions" exec), tool_name ("Bash", "apply_patch", MCP tool names), tool_input:{command:…}, tool_use_id:"call_*"}; PostToolUse adds tool_response (string output). Matchers per tool (e.g. "^Bash$"). Alternative: tail the rollout JSONL — response_item payloads function_call{name:"exec_command",arguments,call_id} / function_call_output{call_id,output} plus reasoning/message items (full fidelity, verified shapes). Caveat: `codex exec --json` docs promise item types command_execution{command,aggregated_output,exit_code,status}, file_change, mcp_tool_call, web_search — but 0.142.5 empirically emitted NO command items (only agent_message + error) while two Bash calls demonstrably ran; don't build on exec-JSON items. app-server streams full item/* lifecycle if ever needed.

*Evidence:* hook-events.jsonl full PreToolUse/PostToolUse payloads; exec2-events.jsonl (missing command items) vs rollout ~/.codex/sessions/2026/07/10/rollout-…019f4cf6….jsonl containing 2 function_call/function_call_output pairs for the same run.

### prompt-capture — full

UserPromptSubmit hook payload has `prompt` (full submitted text) + turn_id + session_id — verified in TUI and exec. Redundant sources: rollout JSONL event_msg user_message{message,images,local_images,text_elements}; ~/.codex/history.jsonl append-only {session_id, ts, text} per user prompt (disable via [history] persistence="none"); notify payload `input-messages` array per turn.

*Evidence:* hook-events.jsonl UserPromptSubmit entries with prompt field; head of ~/.codex/history.jsonl; user_message events in rollout files.

### session-resume — full

Session id = UUIDv7 (time-ordered). Discovery: SessionStart/every hook payload `session_id` + `transcript_path`; exec --json thread.started.thread_id; notify thread-id; rollout filename. Storage: ~/.codex/sessions/YYYY/MM/DD/rollout-<ISO-timestamp>-<uuid>.jsonl (JSONL; first line session_meta{payload.id, cwd, originator:"codex-tui"|exec, cli_version, source, thread_source, model_provider, base_instructions, git}; then turn_context{turn_id, approval_policy, sandbox_policy, permission_profile}, response_item, event_msg lines). Resume: `codex resume <uuid|name> [prompt]` (interactive), `codex resume --last`, picker is cwd-filtered by default (--all, --include-non-interactive); `codex fork <id>|--last` branches; headless `codex exec resume <id|--last> [prompt]`. `--ephemeral` disables session persistence (avoid for vibecast). history.jsonl maps prompts→session_id for forensic recovery.

*Evidence:* help-resume.txt + exec resume help captured; rollout files inspected in ~/.codex/sessions/2026/{06/05,07/10}/; session_meta payload keys listed.

### permissions-guard — full

Layered model: approval_policy = untrusted | on-request (TUI default) | never (exec default) + granular table; sandbox_mode = read-only | workspace-write | danger-full-access (-s flag), sandbox_workspace_write{writable_roots,network_access,…}; `--dangerously-bypass-approvals-and-sandbox` (no --full-auto flag anymore in 0.142.5). External guard: PreToolUse hook can DENY — stdout {"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"…"}} or exit code 2; VERIFIED in TUI: "PreToolUse hook (blocked) / feedback: vibecast guard: rm is blocked on this station", command never ran. PermissionRequest hook can auto-approve/deny approval prompts (docs; untested). execpolicy .rules files (~/.codex/rules/default.rules and project .rules): prefix_rule(pattern=["git","status"], decision="allow") — the TUI "don't ask again" option persists these; --ignore-rules skips them. app-server exposes interactive approval callbacks (item/commandExecution/requestApproval JSON-RPC server→client request). WARNING (verified): after `rm` was denied twice the model deleted the file via `find probe-file.txt -delete` — guards must be semantic, not substring; apply_patch is a separate tool_name to also cover.

*Evidence:* Live deny test in tmux TUI (pane captures + hook-events.jsonl showing rm→deny, apply_patch attempt, find -delete succeeding); ~/.codex/rules/default.rules contents; help texts.

### headless-mode — full

## Recommended integration strategy

(a) Launch: vibecast creates the tmux session and runs `codex -C <workdir> -c developer_instructions="<station system prompt>" "<initial prompt>"` — the positional prompt auto-submits, developer_instructions verifiably appends (not replaces) the system prompt, and AGENTS.md can carry per-repo station docs. Write vibecast's hooks to `<workdir>/.codex/hooks.json` (or a dedicated $CODEX_HOME) and launch with `--dangerously-bypass-hook-trust` to skip the interactive "Hooks need review" gate (accepting two cosmetic warning items), or pre-answer the dialog once via send-keys ("2"+Enter) since trust persists per hook-definition hash and survives edits to the referenced script file. Consider `--no-alt-screen` if the web viewer prefers scrollback-style output. Pre-trust the workdir with `-c 'projects."<dir>".trust_level="trusted"'` so the project hooks layer loads. (b) Event ingestion, ranked: 1) hooks — push-based, fires in the interactive TUI, near-identical payloads to today's Claude Code integration (SessionStart/UserPromptSubmit/PreToolUse/PostToolUse/Stop), so vibecast's existing `hook` subcommand model ports almost 1:1; 2) notify — one-line config for turn-complete with last-assistant-message (argv JSON), ideal completion signal redundancy; 3) tail the rollout JSONL at the transcript_path handed over by the first hook — full fidelity (reasoning, token_count, function_call/output) for the conversation feed; 4) app-server only if vibecast someday wants to own approvals programmatically (it replaces the TUI unless using the experimental `codex --remote` attach); 5) OTEL last — telemetry-grade, not integration-grade. (c) Resume: persist session_id (UUIDv7) from the SessionStart hook (also in notify thread-id and the rollout filename); after a crash relaunch in a fresh pane with `codex resume <uuid>` (optionally with a nudge prompt), or `codex exec resume <uuid>` for headless stations; SessionStart fires again on resume, re-confirming identity. (d) Completion: primary = Stop hook (has last_assistant_message + stop_hook_active); secondary = notify agent-turn-complete; for exec stations, turn.completed + process exit code 0. Session end has no hook — treat codex process/pane exit as session-end. (e) Guard: implement vibecast's dangerous-command policy as a PreToolUse command hook returning permissionDecision deny with a reason (verified blocking in TUI); match on tool_name Bash AND apply_patch AND MCP tools, and evaluate commands semantically — the model demonstrably circumvented a naive `rm` substring block with `find -delete`. Keep sandbox_mode workspace-write + approval_policy on-request as the backstop; never pass --dangerously-bypass-approvals-and-sandbox; optionally auto-answer residual approval dialogs via the PermissionRequest hook or send-keys "y"/"p" with vibecast's own consent flow in front.

## Open questions

- Why did no folder-trust dialog appear on first interactive run in a brand-new untrusted directory on 0.142.5? Need a clean-CODEX_HOME test to map the full first-ever-run onboarding sequence (login screen, consent, trust) that vibecast must auto-answer on fresh machines.
- codex exec --json omitted command_execution/file_change items in 0.142.5 despite current docs promising them — bug or feature-flag gating (item_ids / apply_patch_streaming_events are 'under development')? Re-test on 0.144.1 before relying on exec-JSON for anything beyond turn boundaries.
- Where exactly is hook trust persisted (state_5.sqlite? no sqlite3 in container to confirm), and can vibecast pre-seed it non-interactively instead of using --dangerously-bypass-hook-trust? Alternatively, what is the non-MDM requirements.toml path on Linux for auto-trusted managed hooks?
- PermissionRequest hook request/response shape untested — can it programmatically answer the escalated-approval dialog (the y/p/esc menu), which would let vibecast bridge approvals to its consent service without send-keys?
- apply_patch PreToolUse fired with no matching PostToolUse and no visible block in the deny test — what is the patch tool's failure/approval path, and does PostToolUse fire for it normally?
- app-server daemon + `codex --remote` TUI attach are experimental — stability and protocol churn risk if vibecast ever wants server-owned approvals with the official TUI on top.
- Does notify fire in exec mode too (only verified in TUI; docs imply turn-complete generally)?
- The user's pks-foundry model_provider (loopback token injector) — confirm hooks/notify behave identically when sessions run through it rather than ChatGPT auth.
